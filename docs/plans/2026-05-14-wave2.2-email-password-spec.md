## Goal

Pixel-equivalent port of Laravel's email/password auth flow:
- `/login` (GET + POST)
- `/register` (GET + POST)
- `/forgot-password` (GET + POST)
- `/reset-password/[token]` (GET + POST)
- `/logout` (POST)

All five routes ship in this PR. Each uses Supabase Auth under the hood — no password handling in our code.

---

## Routes + actions

### `/login`
- **GET** — server component renders the login form (email + password). Form posts to a Server Action.
- **POST** (Server Action) — `supabase.auth.signInWithPassword({ email, password })`. On success → `redirect('/borrower')` (or `/admin` based on role; see role-based redirect below). On failure → re-render form with `error` field set.
- Includes a "Forgot password?" link → `/forgot-password`.
- Includes a "Sign up" link → `/register`.
- **Wave 2.3 will add a "Sign in with Solana" button** to this same form; design the layout to leave room (a divider + secondary auth area below the primary email/password block).

### `/register`
- **GET** — server component renders the registration form (email + password + confirm-password).
- **POST** (Server Action) — `supabase.auth.signUp({ email, password })`. The `on_auth_user_created` trigger (from 2.1) auto-creates the `public.users` row with `role='borrower'`. On success → redirect to either the email-verification screen (if Supabase requires email confirm) or `/borrower`.
- Password rules: minimum 12 characters (Supabase default is 6; we override). Reject common patterns (Supabase Auth's leaked-password check is opt-in — leave it for a follow-up).

### `/forgot-password`
- **GET** — single-input form (email).
- **POST** — `supabase.auth.resetPasswordForEmail(email, { redirectTo: '${origin}/reset-password' })`. Always returns a generic "if that email exists, we sent a reset link" message (don't leak whether the email exists).

### `/reset-password/[token]`
- The `[token]` segment captures Supabase's recovery flow token. On GET, the page reads the token from the URL hash via a small client-only fragment (Supabase's recovery flow puts the token in the URL fragment, which the server doesn't see). Once the session exchange completes, render the new-password form.
- **POST** (Server Action) — `supabase.auth.updateUser({ password: newPassword })`. On success → redirect `/login` with a success banner.

### `/logout`
- **POST only** — Server Action calls `supabase.auth.signOut()` then redirects to `/`. GET requests to `/logout` are 405 — CSRF prevention.

## Role-based post-login redirect

After `/login` succeeds, redirect by role:
- `role = admin` → `/admin`
- `role = evaluator_online` or `role = evaluator_offline` → `/evaluator`
- `role = borrower` → `/borrower` (or `/borrower/onboarding` if `public.users.solana_address is null` — soft nudge to link a wallet)

Implementation: after `signInWithPassword`, call `getServerUser()` (the 2.1 helper) to read the role, then `redirect()` to the appropriate path.

Note: `/borrower`, `/evaluator`, `/admin` routes don't exist yet (Waves 3–6). For Wave 2.2, redirect to `/` (the home page) as a placeholder, with a TODO comment pointing at this section of the spec.

## Acceptance criteria

1. All 5 routes render their Laravel-equivalent forms (pixel-equivalent at 1440px desktop and 375px mobile)
2. Login with valid credentials sets a Supabase session cookie and redirects per role rules
3. Login with bad credentials re-renders with an error (no info leak about whether email exists)
4. Registration creates an `auth.users` row → trigger creates `public.users` row with `role='borrower'`
5. Forgot-password sends a reset email via Supabase (verifiable in Supabase dashboard Auth → Users)
6. Reset-password with a valid token updates the password
7. Logout clears the session and redirects to `/`
8. `pnpm -w turbo run lint typecheck test build --filter=@vaulx/web` clean
9. Vitest covers happy paths and key failure modes for each route
10. No service-role key usage in this PR — anon key + session cookie only

## Security reviewer brief

> Review `feat/auth-email-password`. Focus:
> 1. Password handling — credentials NEVER touch our code beyond passing to `signInWithPassword` / `signUp` / `updateUser`. Never logged, never stored, never echoed in error responses.
> 2. Session cookie attributes — verify `@supabase/ssr` sets HttpOnly, SameSite=Lax (or stricter), Secure in production. Inspect any `cookies().set()` calls.
> 3. CSRF — Server Actions have automatic Origin/Referer checking in Next 14+. Verify no GET-triggered state changes. The `/logout` route MUST be POST-only.
> 4. Reset-password token — verify it's read from URL fragment client-side and exchanged via Supabase, not naively stored or echoed.
> 5. Error responses — no info leaks ("user not found" vs "wrong password" both return same generic message).
> 6. Rate limiting — Supabase Auth has built-in rate limits on signIn/signUp/reset. Verify we don't bypass them.
> 7. Email enumeration on `/forgot-password` — response must be generic regardless of whether email exists.
> 8. No service-role key usage.
> 9. No secrets in diff.
