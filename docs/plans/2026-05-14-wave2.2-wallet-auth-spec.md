## Goal

A single auth modal that lets any visitor sign in to Vaulx by **controlling a Solana wallet** — whether they already have one (Phantom) or Crossmint creates one for them on the fly.

After this PR:
- Anonymous visitor can click "Sign in" anywhere on the site
- Modal opens with two choices:
  - "Continue with email or social" → Crossmint widget → email/Google/Apple → wallet → sign-message → session
  - "I already have a wallet" → wallet-adapter modal (Phantom/Solflare/Backpack/etc.) → sign-message → session
- Either path ends with: Supabase Auth session cookie set + `public.users` row created/updated with the wallet pubkey + email (Crossmint provides the email; direct-SIWS users have a synthetic email matching Laravel's `<pubkey>@siws.vaulx.local` pattern)
- "Sign out" works from any page header

**No traditional email/password forms at all.** No `/login`, `/register`, `/forgot-password`, `/reset-password` as separate routes.

## Auth flow — concrete

### Crossmint path
1. User clicks "Sign in" → modal opens
2. User picks "Continue with email or social"
3. Crossmint widget opens → user authenticates with email/Google/Apple
4. Crossmint provisions (or unlocks) the user's smart wallet → we get `publicKey`
5. Crossmint surfaces the user's email via `useAuth()` (or equivalent SDK hook)
6. Client: build canonical message `vaulx:auth:<timestamp>:<pubkey>`, sign via `useUnifiedWallet().signMessage`
7. Client: call Supabase `auth.signInWithWeb3({ chain: 'solana', wallet: publicKey, signature, message })`
8. Supabase verifies the signature, finds or creates an `auth.users` row (synthetic email or Crossmint email — see "User-row strategy" below), issues a session JWT, sets the cookie
9. Trigger from 2.1 creates `public.users` row if new; otherwise existing row is reused
10. Client-side: call a Server Action `linkCrossmintEmail(email)` that updates `public.users.email` (and `public.users.solana_address` if missing) for the just-authenticated user, gated by RLS so users can only update their own row

### Direct-SIWS path
1. User clicks "Sign in" → modal opens
2. User picks "I already have a wallet" → wallet-adapter modal
3. User picks Phantom/Solflare/etc. → connects
4. Steps 6–9 from the Crossmint path are identical (the signature flow is the same)
5. `public.users.email` is set to the synthetic `<pubkey>@siws.vaulx.local` (matching Laravel)

## User-row strategy

Supabase Auth's `signInWithWeb3` for Solana:
- Stores the wallet pubkey in `auth.users.app_metadata.wallet` (per Supabase SIWS docs — verify via Context7)
- Creates the `auth.users` row with `email` set to a Supabase-generated placeholder OR a provided email
- Our trigger `on_auth_user_created` (from 2.1) inserts a matching `public.users` row with `role='borrower'` and `email = new.email` and **`solana_address = null`**

Then our **post-signin Server Action** (`linkCrossmintEmail` for Crossmint, `linkWalletEmail` for direct) populates `public.users.solana_address` from the just-authenticated wallet pubkey, and updates `email` if Crossmint provided a real one. RLS on `public.users` already ensures the action only mutates the caller's own row.

**The synthetic email pattern for direct-SIWS users:** since Supabase Auth requires an email (even a placeholder), we use `<pubkey>@siws.vaulx.local` matching Laravel's pattern (per `docs/plans/inventory/03-auth-and-siws.md`). The Crossmint path uses Crossmint's email.

## Acceptance criteria

1. "Sign in" button in MarketingNav opens the auth modal
2. Crossmint path: clicking it opens Crossmint widget, user picks email/Google/Apple, wallet is provisioned, message is signed, Supabase session is established, `public.users` row exists with the user's Crossmint email and wallet
3. Direct-SIWS path: clicking it opens wallet-adapter modal, user picks a wallet, message is signed, Supabase session is established, `public.users` row exists with `<pubkey>@siws.vaulx.local` and the wallet
4. After sign-in, MarketingNav shows the user-menu (display name or short pubkey, dropdown with "Sign out")
5. "Sign out" clears the session, returns to `/`
6. `public.users.solana_address` is uniquely populated — two users can't claim the same wallet (enforced by 2.1's unique constraint)
7. `pnpm -w turbo run lint typecheck test build --filter=@vaulx/web` clean
8. Vitest covers the happy paths + auth-required failure cases
