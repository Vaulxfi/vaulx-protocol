This is the first of four Wave 2 PRs. It adds **no user-facing routes** — only the data layer, RLS, hooks, and trigger that every subsequent Wave 2 PR consumes. Keeping it foundation-only makes the diff small and the security review tight.

## Goal

Establish `public.users` as the Vaulx app's user-data table in Supabase Postgres, with proper RLS, an `auth.users → public.users` trigger, and TS server/client helpers. After this PR merges, the codebase has the infrastructure to read "who is the current user" — but no UI yet that uses it.

## Files to add

### 1. Supabase migration: `supabase/migrations/20260515090000_create_public_users.sql`

```sql
-- public.users — Vaulx app-level user data, 1:1 with auth.users.
--
-- Pattern: https://supabase.com/docs/guides/auth/managing-user-data
-- Future tables (loans, assets, evaluations) FK to public.users.id.

begin;

create type public.user_role as enum (
  'borrower',
  'admin',
  'evaluator_online',
  'evaluator_offline'
);

create table public.users (
  id uuid primary key references auth.users(id) on delete cascade,
  email text not null,
  role public.user_role not null default 'borrower',
  display_name text,
  solana_address text unique,            -- b58 pubkey, set via SIWS in Wave 2.3
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index users_role_idx on public.users (role);
create index users_solana_address_idx on public.users (solana_address)
  where solana_address is not null;

-- Trigger: keep public.users.email in sync with auth.users.email
create or replace function public.handle_auth_user_email_change()
returns trigger language plpgsql security definer set search_path = public as $$
begin
  update public.users set email = new.email, updated_at = now()
    where id = new.id;
  return new;
end;
$$;

create trigger on_auth_user_email_changed
  after update of email on auth.users
  for each row execute function public.handle_auth_user_email_change();

-- Trigger: auto-create public.users row when auth.users is created
create or replace function public.handle_new_auth_user()
returns trigger language plpgsql security definer set search_path = public as $$
begin
  insert into public.users (id, email, role)
  values (new.id, new.email, 'borrower');
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_auth_user();

-- RLS
alter table public.users enable row level security;

-- Each user can read + update their own row
create policy users_self_read on public.users
  for select using (auth.uid() = id);

create policy users_self_update on public.users
  for update using (auth.uid() = id)
  with check (auth.uid() = id);

-- Admins can read all rows (for the future admin portal in Wave 6)
create policy users_admin_read on public.users
  for select using (
    exists (
      select 1 from public.users u
      where u.id = auth.uid() and u.role = 'admin'
    )
  );

-- updated_at touch trigger
create or replace function public.touch_updated_at()
returns trigger language plpgsql as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

create trigger users_touch_updated_at
  before update on public.users
  for each row execute function public.touch_updated_at();

commit;
```

Notes for the integrator:
- The `auth.users` schema is Supabase-managed. Triggers ON `auth.users` require the migration to be run with sufficient privileges; Supabase's default service-role JWT during local development supports this. If the migration fails on `auth.users` permissions, document the workaround (run via Supabase dashboard SQL editor) and flag it.
- The `users_admin_read` policy creates a small RLS recursion risk if not careful. Verify it works correctly under `auth.uid() = NULL` (no session) — should return zero rows.
- `solana_address` index is partial (`where solana_address is not null`) to keep the index small until many users have linked wallets.

### 2. `apps/web/src/lib/auth/server.ts` — server-side helpers

Provide three exports:

```ts
// Returns the authenticated user row from public.users, or null.
// Uses the Supabase server-side SSR client; reads the session cookie.
export async function getServerUser(): Promise<PublicUser | null>;

// Same as getServerUser but throws / redirects if no session.
// For use in server components and route handlers that must be authenticated.
export async function requireUser(): Promise<PublicUser>;

// Helper for route handlers — returns the user with a typed Supabase client
// for further queries scoped to that user's RLS context.
export async function getAuthedClient(): Promise<{
  user: PublicUser;
  supabase: SupabaseClient;
}>;
```

Use `@supabase/ssr` (already a recommended dep per Next App Router auth docs — check Context7). The integrator must verify it's in `apps/web/package.json` or `packages/supabase`; if not, add it (this is the ONE allowed new dep for this wave).

Type `PublicUser` lives in `apps/web/src/lib/auth/types.ts`:

```ts
export type UserRole = 'borrower' | 'admin' | 'evaluator_online' | 'evaluator_offline';

export type PublicUser = {
  id: string;
  email: string;
  role: UserRole;
  display_name: string | null;
  solana_address: string | null;
  created_at: string;
  updated_at: string;
};
```

## Acceptance criteria

1. Supabase migration applies cleanly against a fresh local Supabase. The `auth.users` trigger works (manual test: insert into `auth.users` via SQL → `public.users` row appears with role=borrower).
2. RLS policies pass these tests:
   - User A reads their own row → ok
   - User A reads User B's row → returns zero rows
   - Admin user reads any row → ok
   - Anonymous (no session) reads anything → returns zero rows
3. `getServerUser()` / `requireUser()` / `getAuthedClient()` exported and typed correctly.
4. `useUser()` stub returns `{ user: null, loading: false }` and is callable from any client component.
5. `pnpm -w turbo run lint typecheck test build --filter=@vaulx/web` clean.
6. No new npm deps **except** `@supabase/ssr` if not already present.
7. No edits to `site/`, `apps/web/src/app/demo/*`, `globals.css`, `tailwind.config.ts`, or any sibling-wave work.

## Security reviewer brief

> Review `feat/auth-supabase-foundation` against this spec. Focus:
> 1. RLS coverage on `public.users`: a non-admin user must not be able to read/update another user's row, including via API edge cases.
> 2. The `auth.users` triggers run as `security definer` — verify the search_path is locked to `public` to prevent privilege escalation via a malicious schema.
> 3. The `users_admin_read` policy's subquery — does it cause infinite recursion if RLS is enabled on the subquery's own `public.users` read? Test by trying SELECT as a non-admin.
> 4. `solana_address` unique index — duplicate-wallet handling. Two users can't claim the same wallet.
> 5. Service-role usage in `getAuthedClient()` — must NOT bypass RLS unintentionally; should use the user's JWT, not service-role.
> 6. `requireUser()` failure mode — does it leak service-role state in error responses?
> 7. No secrets in diff.
>
